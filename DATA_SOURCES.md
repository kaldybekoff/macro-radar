# Источники и сбор данных Macro Radar Kazakhstan

## Общая схема

Данные не копируются вручную и не вводятся через интерфейс сайта. Программа напрямую обращается к внешним источникам, извлекает показатели, проверяет их и сохраняет в историческую базу Supabase.

Этот процесс можно описать как ETL:

```text
Extract — получить данные из источника
Transform — привести их к единому формату и проверить
Load — сохранить в базу
```

Полный процесс выглядит так:

```text
GitHub Actions или кнопка «Обновить данные»
                       ↓
       Запросы к НБРК, БНС и Yahoo Finance
                       ↓
          Извлечение значений и дат
                       ↓
       Приведение к единому формату
                       ↓
          Проверка допустимых значений
                       ↓
             Запись в Supabase
                       ↓
       Отображение на сайте и экспорт
```

## Краткая таблица источников

| Показатель | Источник | Формат | Способ извлечения |
|---|---|---|---|
| USD/KZT | Национальный Банк Казахстана | XML/RSS | Поиск валюты `USD` и чтение официального курса |
| Базовая ставка | Репозиторий открытых данных НБРК | JSON API | Чтение поля `baseRate` |
| TONIA | Репозиторий открытых данных НБРК | JSON API | Чтение поля `tonia` |
| Инфляция и её компоненты | Бюро национальной статистики | HTML-публикация | Поиск последней публикации и извлечение показателей из текста |
| Brent | Yahoo Finance | Рыночный временной ряд | Последняя доступная цена закрытия фьючерса `BZ=F` |

## 1. Курс USD/KZT

### Источник

Официальный RSS Национального Банка Республики Казахстан:

```text
https://nationalbank.kz/rss/get_rates.cfm?fdate=ДД.ММ.ГГГГ
```

Например:

```text
https://nationalbank.kz/rss/get_rates.cfm?fdate=15.07.2026
```

### Как извлекается значение

НБРК возвращает XML со списком валют. Фрагмент ответа может выглядеть так:

```xml
<item>
    <title>USD</title>
    <description>468.88</description>
</item>
```

Программа:

1. Загружает XML.
2. Перебирает элементы списка валют.
3. Находит элемент с кодом `USD`.
4. Читает значение из поля `description`.
5. Заменяет запятую на точку, если это необходимо.
6. Преобразует значение в число.
7. Сохраняет его под кодом `USD_KZT`.

Это официальный курс НБРК, а не курс покупки или продажи конкретного банка.

### Ограничение

В текущей реализации периодом записи является дата запуска загрузки. В выходной или праздничный день официальный ответ может фактически относиться к ближайшему рабочему дню. В будущем желательно дополнительно считывать официальную дату непосредственно из ответа НБРК.

## 2. Базовая ставка и TONIA

### Источник

Публичный API Репозитория открытых данных НБРК:

```text
https://data.nationalbank.kz/api/indicators/latest
```

API возвращает структурированный JSON, например:

```json
{
  "date": "2026-07-15",
  "baseRate": 17.00,
  "tonia": 16.79
}
```

### Как извлекаются значения

Программа читает три поля:

- `date` — дата показателей;
- `baseRate` — базовая ставка;
- `tonia` — ставка TONIA.

После этого создаются две отдельные записи:

```text
NBK_BASE_RATE = 17.00%
TONIA = 16.79%
```

Получение через JSON API является наиболее устойчивым способом сбора в текущем проекте: программа читает готовые поля, а не распознаёт текст web-страницы.

### Аварийные значения

Параметры:

```text
MANUAL_BASE_RATE
MANUAL_TONIA_RATE
```

являются необязательным аварийным резервом. Они используются только тогда, когда API НБРК недоступен или вернул некорректный ответ. При нормальной работе автоматические данные НБРК всегда имеют приоритет.

Если ручные значения не заданы и API временно недоступен, новая запись не создаётся, но последнее успешно сохранённое значение остаётся в Supabase и продолжает отображаться с предупреждением о свежести.

## 3. Инфляция и её компоненты

### Источник

Раздел «Цены» Бюро национальной статистики:

```text
https://stat.gov.kz/ru/industries/economy/prices/
```

### Как извлекаются значения

Здесь используется разбор HTML-страниц, то есть web scraping. Процесс состоит из двух запросов.

Сначала программа:

1. Открывает раздел «Цены».
2. Просматривает ссылки на странице.
3. Ищет ссылку, в названии которой есть фраза «Инфляция в Республике Казахстан».
4. Выбирает публикацию и переходит по её адресу.

Затем программа:

1. Загружает HTML выбранной публикации.
2. Удаляет разметку и получает обычный текст.
3. Находит месяц и год публикации.
4. Ищет в тексте числовые значения по характерным формулировкам.

Например, из текста:

```text
Инфляция в Республике Казахстан в июне 2026 года составила 10,3%,
за месяц — 0,7%.
```

извлекаются:

- период — июнь 2026 года;
- годовая инфляция — 10,3%;
- месячная инфляция — 0,7%.

Также программа пытается получить:

- продовольственную инфляцию;
- непродовольственную инфляцию;
- инфляцию платных услуг.

Период и годовая инфляция обязательны. Остальные компоненты являются дополнительными: если один компонент не найден, найденные показатели всё равно могут быть сохранены.

### Ограничение

Это наиболее уязвимый способ extraction в текущем проекте. Если БНС изменит заголовок, порядок ссылок или формулировки публикации, автоматический разбор может перестать находить необходимые значения.

Для повышения надёжности в будущем желательно перейти на официальный API БНС или другой структурированный источник.

## 4. Цена нефти Brent

### Источник

Yahoo Finance, тикер ближайшего фьючерса Brent:

```text
BZ=F
```

Страница инструмента:

```text
https://finance.yahoo.com/quote/BZ=F/
```

### Как извлекается значение

Программа через библиотеку `yfinance`:

1. Запрашивает историю за последние 10 дней.
2. Выбирает столбец `Close` — цену закрытия.
3. Удаляет пустые значения.
4. Находит последнюю доступную запись.
5. Округляет цену до двух знаков.
6. Сохраняет дату и значение в долларах США за баррель.

### Ограничение

`BZ=F` — рыночная котировка ближайшего фьючерсного контракта Brent, а не официальная статистическая или физическая спотовая цена нефти.

При переходе рынка с одного фьючерсного контракта на следующий в ряду могут возникать изменения, связанные не только с движением нефтяного рынка, но и с разницей цен между контрактами. Для рабочего макромониторинга такой источник подходит как ориентир, однако его нельзя считать официальной статистикой.

## Приведение данных к единому формату

Каждое извлечённое значение превращается в стандартное наблюдение. Например:

```text
Код показателя: TONIA
Период: 2026-07-15
Значение: 16.79
Единица измерения: %
Источник: NBK
Частота: daily
Дата загрузки: время выполнения обновления
Тип данных: fact
Статус качества: valid
```

Основные поля:

- `indicator_code` — внутренний код показателя;
- `period` — дата или месяц, к которому относится значение;
- `value` — числовое значение;
- `unit` — единица измерения;
- `source` — источник;
- `source_url` — ссылка на источник;
- `frequency` — периодичность;
- `fetched_at` — время загрузки;
- `quality_status` — результат проверки.

Единый формат позволяет хранить показатели из разных источников в одной таблице и одинаково использовать их в графиках, сравнениях и отчётах.

## Проверка значений

Перед сохранением программа проверяет, что показатель находится в допустимом диапазоне:

| Показатель | Допустимый диапазон |
|---|---:|
| USD/KZT | от 100 до 1 500 |
| Годовая инфляция | от −10% до 100% |
| Месячная инфляция | от −20% до 30% |
| Компоненты инфляции | от −20% до 150% |
| Brent | от 5 до 300 USD/bbl |
| Базовая ставка | от 0% до 50% |
| TONIA | от 0% до 100% |

Такая проверка отсекает очевидные технические ошибки. Например, значение базовой ставки `1700` вместо `17.00` не будет записано.

Проверка диапазона не подтверждает экономическую правильность показателя. Она защищает только от явных аномалий формата или масштаба.

## Сохранение в Supabase

После проверки данные записываются в таблицу `observations`.

Используется операция `upsert`:

- если записи ещё нет, она создаётся;
- если запись с теми же ключевыми полями существует, она обновляется;
- повторный запуск не создаёт дубликат.

Уникальность определяется сочетанием:

```text
показатель + период + источник + тип данных
```

Например, два запуска в один день не должны создать две одинаковые записи TONIA от НБРК за одну дату.

Более 4 000 исторических наблюдений были первоначально перенесены из старого Excel. Новые значения добавляются из внешних источников автоматически.

## Обработка ошибок

Каждый источник обрабатывается независимо.

Если, например, публикация БНС недоступна:

- курс USD/KZT может обновиться;
- базовая ставка и TONIA могут обновиться;
- Brent может обновиться;
- ошибка БНС записывается в журнал;
- последнее сохранённое значение инфляции остаётся в базе.

Статусы запуска:

- `success` — все доступные источники обработаны успешно;
- `partial` — часть источников обновилась, часть завершилась ошибкой;
- `failed` — не удалось получить ни одного нового наблюдения.

История не удаляется при ошибке источника.

## Расписание

GitHub Actions запускает сбор автоматически по рабочим дням в:

```text
08:30, часовой пояс Asia/Qyzylorda
```

Обновление можно также запустить:

- кнопкой «Обновить данные» на сайте;
- вручную в GitHub через `Actions → Update macro data → Run workflow`.

Все варианты используют одни и те же коннекторы, проверки и правила сохранения.

## Относительная надёжность источников

По устойчивости текущие способы сбора можно расположить так:

1. JSON API НБРК — структурированные поля, наиболее надёжный вариант.
2. XML/RSS НБРК — официальный и структурированный источник.
3. Yahoo Finance — удобный рыночный источник, но не официальная статистика.
4. Разбор HTML-публикации БНС — официальный первоисточник, но технически наиболее чувствительный к изменениям сайта.

## Что желательно улучшить дальше

1. Получать инфляцию через официальный структурированный API или файл БНС.
2. Извлекать официальную дату курса USD/KZT из ответа НБРК.
3. Добавить повторные попытки запроса при временных сетевых ошибках.
4. Записывать отдельные предупреждения при неожиданном изменении структуры ответа.
5. Рассмотреть резервный источник котировки Brent.
6. Добавить автоматическую сверку последних значений с альтернативным источником.

## Итог

Macro Radar собирает данные четырьмя способами: JSON API, XML/RSS, разбор HTML-публикации и получение рыночного временного ряда. После extraction все показатели приводятся к одному формату, проверяются и сохраняются в Supabase без дублирования.

Система сокращает ручную работу, но не отменяет контроль источника, даты публикации и экономического смысла показателей со стороны аналитика.
