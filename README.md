# Macro Radar Kazakhstan

Личный инструмент для автоматического мониторинга макроэкономических показателей Казахстана. Приложение собирает данные из открытых источников, сохраняет историю, показывает изменения на dashboard и формирует Excel- и PowerPoint-отчёты.

## Проблема

Подготовка регулярного макроэкономического обзора обычно состоит из повторяющихся операций:

- открыть несколько сайтов и найти последние публикации;
- вручную перенести показатели в Excel;
- проверить даты и единицы измерения;
- сравнить значения с предыдущим периодом;
- перестроить графики;
- обновить таблицы и презентацию;
- повторить весь процесс через неделю.

Такой подход занимает время, создаёт дубликаты и повышает риск использовать устаревшее или неправильно интерпретированное значение. Особенно сложно поддерживать единую историю, когда дневные, месячные и рыночные показатели обновляются с разной периодичностью.

Macro Radar решает эту задачу как небольшой персональный ETL- и reporting-инструмент: сбор, проверка, хранение, визуализация и экспорт объединены в одном воспроизводимом процессе.

## Подход

Система разделена на независимые слои:

```text
НБРК / БНС / Yahoo Finance
             ↓
      Python-коннекторы
             ↓
    Валидация и нормализация
             ↓
       Supabase/PostgreSQL
             ↓
   Аналитика и сравнения
             ↓
 Streamlit / Excel / PowerPoint
```

### Сбор данных

- `USD/KZT` загружается из официального RSS НБРК.
- Инфляция и её компоненты извлекаются из последней публикации БНС.
- Brent загружается по рыночному тикеру `BZ=F`.
- Базовая ставка и TONIA вынесены в отдельный коннектор. При недоступности автоматического источника поддерживаются явно заданные резервные значения.

Каждый источник работает независимо. Если один из них недоступен, успешные данные остальных источников всё равно сохраняются, а запуск получает статус `partial`.

### Хранение

Supabase используется как постоянная историческая база. Основная таблица `observations` хранит:

- код показателя;
- период, к которому относится значение;
- значение и единицу измерения;
- источник и ссылку;
- частоту данных;
- тип `fact/forecast`;
- время получения;
- статус качества.

Уникальное ограничение и операция `upsert` защищают историю от дублей. Старые Excel/CSV-файлы используются только для однократной миграции и больше не являются источником истины.

### Надёжность данных

- Проверяются допустимые диапазоны показателей.
- Официальные источники имеют приоритет над импортированными legacy-снимками.
- Для CPI используется официальный отчётный месяц, а не дата запуска приложения.
- При сбое dashboard продолжает показывать последнее успешное значение.
- Устаревшие показатели помечаются предупреждением.
- Каждый запуск записывается в `ingestion_runs` со статусом `success`, `partial` или `failed`.

### Автоматизация

GitHub Actions запускает обновление по рабочим дням в `08:30 Asia/Qyzylorda`. Обновление также можно запустить вручную из GitHub или через кнопку в Streamlit.

Streamlit Community Cloud отвечает только за интерфейс. История не хранится на временной файловой системе Streamlit и не теряется при перезапуске приложения.

## Результат

Реализован работающий сквозной процесс:

```text
Получение данных
→ проверка
→ запись в Supabase
→ расчёт изменений
→ dashboard и графики
→ Excel/PowerPoint по запросу
```

Достигнутые результаты:

- перенесено более 4 000 исторических наблюдений;
- настроено ежедневное автоматическое обновление;
- исключено дублирование одинаковых наблюдений;
- сохранение истории не зависит от доступности Streamlit;
- сбой одного источника не блокирует весь pipeline;
- рассчитываются изменения к предыдущему наблюдению, за неделю и за месяц;
- доступны интерактивные графики USD/KZT, Brent и инфляции;
- Excel и PowerPoint создаются непосредственно из базы;
- dashboard развёрнут в Streamlit Community Cloud;
- автоматизированный workflow развёрнут в GitHub Actions;
- ключевая логика покрыта тестами.

В результате ручной процесс сводится к проверке свежести данных, просмотру сигналов и скачиванию готового отчёта.

## Настройка Supabase

1. Создайте проект Supabase.
2. Откройте `SQL Editor`.
3. Выполните [`supabase/schema.sql`](supabase/schema.sql).
4. В `Project Settings → API Keys` получите Project URL и Secret key.

RLS включён для всех прикладных таблиц. Публичные политики намеренно отсутствуют: запись и чтение выполняются серверной частью с Secret key.

Никогда не добавляйте Secret key в Git, README, issue или клиентский JavaScript.

## Локальный запуск

```powershell
py -3.12 -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements-dev.txt
Copy-Item .streamlit\secrets.toml.example .streamlit\secrets.toml
```

Заполните `.streamlit/secrets.toml`:

```toml
SUPABASE_URL = "https://your-project.supabase.co"
SUPABASE_SERVICE_ROLE_KEY = "your-secret-key"

# Необязательный резервный источник ставок
MANUAL_BASE_RATE = 17.0
MANUAL_TONIA_RATE = 17.11
```

Запустите миграцию старых данных при первом развёртывании:

```powershell
python scripts/migrate_legacy.py
```

Запустите приложение:

```powershell
streamlit run app.py
```

Локальный адрес: `http://localhost:8501`.

## GitHub Actions

В `Settings → Secrets and variables → Actions` создайте:

- `SUPABASE_URL`;
- `SUPABASE_SERVICE_ROLE_KEY`;
- `MANUAL_BASE_RATE` — опционально;
- `MANUAL_TONIA_RATE` — опционально.

Ручной запуск:

```text
Actions → Update macro data → Run workflow
```

## Streamlit Community Cloud

При создании приложения укажите:

```text
Repository: kaldybekoff/macro-radar
Branch: main
Main file: app.py
Python: 3.12
```

В `Settings → Secrets` добавьте те же переменные Supabase. GitHub Actions Secrets и Streamlit Secrets являются разными хранилищами и настраиваются отдельно.

## Тесты

```powershell
pytest
```

Тестируются парсеры источников, валидация, частично успешные запуски, сравнения и приоритет официальных данных над legacy-записями.

## Ограничения

- Структура сайтов-источников может измениться, поэтому парсеры требуют мониторинга.
- Yahoo Finance является рыночным, а не официальным статистическим источником.
- Резервные значения базовой ставки и TONIA необходимо обновлять после решений НБРК.
- Автоматические выводы являются вспомогательными и не заменяют аналитическую проверку.
